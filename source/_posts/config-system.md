配置系统可说是开发者最熟悉的陌生人。从业务功能开关到服务监听端口，从系统调度策略到数据缓冲区大小，每个应用运行起来都需要开发者显式或隐式地指定大量配置。

然而，很少有开发者认真思考过配置系统是如何工作的，好像应用天然就能自己加载到正确的配置。一旦应用加载到错误的配置，开发者则往往很难说出问题出在哪里。要么怪罪配置有多个来源造成混乱，要么反思配置项的设计不够合理，要么指责配置的管理职责不明，莫衷一是。

在我看来，难以厘清配置系统的问题，其根源在于开发者在谈及配置的时候，经常把多个维度的问题关注点混合起来讨论，导致各说各话，还容易把不相干的问题牵扯进来一锅乱炖。

## 配置的定义和分类

我们首先给配置下一个定义：**配置是一种无需修改源码就能改变应用行为的数据。**

以业务功能开关为例，应用代码中同时存在开关开启和关闭的执行路径，开发者通过各种方式让应用运行时加载到开关配置值，根据配置值指示功能开启或关闭，执行对应的业务逻辑。

又例如，对外公开的应用服务启动时，都需要配置监听外部请求的端口，同时其他应用访问该服务时，也需要知道它的监听端口，才能保证服务网络正常通信。

这个配置定义涵盖的内容非常广泛。常见的配置分类方式，可能关注配置的来源、是否必要、如何共享、是否加密等问题。这个关注点当然有各自的道理，也都导向一些具体的技术方案。然而，对于线上业务来说，提纲挈领的分类标准只有一个：**这是一个业务配置，还是一个应用配置？**

业务配置指的是独立于应用实现，只影响业务逻辑的配置。例如上面提到的业务功能开关，同类活动可配置用户标签，A/B 测试的灰度策略等等。

应用配置则指的是控制应用程序自身运行，无关业务逻辑的配置。例如上面举例的服务端口，数据库连接串，系统日志的输出等级等等。

现实世界当中，业务逻辑实现在应用程序上，所以业务配置和应用配置最终都由应用程序加载，共同决定应用程序各个维度上的实际行为。现有的配置软件大多基于技术实现定义自己的领域范围，很少直观区分这两种配置，以至于在开发者之间讨论的时候，经常被技术实现误导，将两种配置混在一起谈。

## 配置软件的分类

基本上，配置软件的主要关注点总是落在以下三点之内：

1. 如何书写配置
2. 如何加载配置
3. 如何管理配置

### 配置语言定义如何书写配置

最简单的配置表达方式是字符串键值对。例如通过环境变量配置连接串，可以写成 `export SCOPEDB_ENDPOINT="https://tenant-name.scopedb.cloud/"` 的形状。

进一步地，要想表达层次结构的配置，可以采用 JSON、YAML、TOML 等格式。形如：

```yaml
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-cloudflare
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: account@company.com
    privateKeySecretRef:
      name: letsencrypt-cloudflare-account-key
    solvers:
      - dns01:
          cloudflare:
            apiTokenSecretRef:
              name: <cloudflare-api-token-secret>
              key: api-token
```

更进一步，如果想要在一个配置项中引用另一个配置项，或者高度自定义某一配置项如何组装，可以使用 HOCON、CUE 或 Jsonnet 这样的复杂配置语言。以下是一个 Jsonnet 配置的例子：

```lua
local application = 'my-app';
local module = 'uwsgi_module';
local dir = '/var/www';
local permission = 644;

{
  'uwsgi.ini': std.manifestIni({
    sections: {
      uwsgi: {
        module: module,
        pythonpath: dir,
        socket: dir + '/uwsgi.sock',
        'chmod-socket': permission,
        callable: application,
        logto: '/var/log/uwsgi/uwsgi.log',
      },
    },
  }),
}
```

总的来说，配置语言是配置的表述形式。最常见的是字符串键值对。中等复杂系统开始会使用有层次结构的配置。复杂配置语言首要解决的是一个配置引用另一个的问题，其他具体实现里长出来的数据验证，乃至接近完整功能的通用编程语言的能力，则是各自发展过程中杂糅进来的需求。

### 进程如何加载配置

第二类配置软件解决进程如何加载配置的问题。

不同编程语言生态都会有这样的软件库，但是往往并不非常流行，因为常见的做法是每个应用自己定义进程如何加载配置值。这里首先罗列不同语言生态当中的参考实现：

* Go 语言的实现算是最流行的，其名为 [Viper](https://github.com/spf13/viper)
* Rust 语言的实现用户也不少，叫做 [config-rs](https://github.com/rust-cli/config-rs)
* Ruby 语言的实现随着 Ruby on Rails 一起广泛传播，可能是第一个定义此类软件形态的实现，就叫 [config](https://github.com/rubyconfig/config)
* Java 语言也有类似的实现，但是知名度就要低许多，名字是 [cfg4j](https://github.com/cfg4j/cfg4j)
* 当然，Java 生态里更常见的是 Spring Boot 内嵌的配置加载逻辑。

所有这类软件解决的都是同样的两个核心问题：

1. 配置信息从哪里获取？
2. 相同路径的配置项，以哪个为准？

可以看到，上面所举的示例软件，大多支持环境变量、命令行参数、不同格式的配置文件、远程配置服务等配置信息来源。在软件内部，它们都会将所有加载上来的配置信息解析成统一的层次配置结构。对于相同路径的配置项，这些软件大多有预先定义的加载优先级，部分软件可以自定义加载优先级。

现实世界当中，配置来源千奇百怪，加载优先级跟具体技术实现应用需求强挂钩。上面介绍的开箱即用方案往往不能很好的理解应用需求，导致其预制逻辑经常出乎开发者预料。再加上从不同来源读取配置信息的库往往是现成的，实现一个分层加载方案并不复杂，因此，大部分应用服务会绕过这类配置库，直接内联实现配置加载逻辑。但是其解决的问题是不变的。

### 配置的环境和权限管理

最后一类配置软件关注配置的环境和权限管理。

由于其功能的复杂性，这类软件往往会实现会以独立进程的形式运行，并提供 SDK 给应用软件执行管理操作。

[Apollo](https://www.apolloconfig.com/) 配置管理系统是这类软件的典型案例。它集中管理了不同业务在不同环境里的配置信息，其领域模型大致是：

* 一个组织可以有多个团队和成员，一个成员可以属于若干个团队
* 一个组织可以有多个服务，每个服务对应到一个线上业务或微服务
* 一个服务可以有多个环境，每个环境各自有一套配置信息

对于权限管理，则有两个维度：

* 某个成员是否有某个服务在某个环境下的所有配置的读或写权限
* 某个服务是否有读另一个服务在相同环境下的配置的权限

其他配置管理系统也基本大同小异。

## 配置的领域模型映射到技术实现

现在，我们回过头来看为什么配置的主要分类标准是业务配置/应用配置。

核心原因是它区分了配置管理的责任。**对于业务配置来说，有权修改配置的是产品和运营；而对于应用配置来说，有权修改配置的是研发和运维。**

当然，实际组织运行的情况，研发有可能作为产品和运营的代理执行配置修改动作，DevOps 可能是一个人同时担任，甚至小团队里面，所有的职责可能都对应到同一个人。但是这不影响软件工程角度下的责权划分。

明确责任以后，具体的技术选型反而是非常灵活的。当然，业务配置和应用配置的技术实现有一定的倾向性，但都不是百分之百。

例如，就配置语言而言，业务配置往往是简单的键值对，很少有层次结构，而应用配置经常有层次结构。对于配置校验，业务配置的校验往往写在代码里随新版本发布，而应用配置则可以从校验工具（CUE/JSON Schema 等）收益。

配置值的来源，业务配置通常来自独立配置系统，也更容易有配置热更新的需求，而应用配置则更可能来自配置文件、命令行参数、环境变量等，且通常在启动后就不再变更。

配置管理和权限问题，就是一开始提到的责权分配了。

## 为何关于配置的讨论总是牛头不对马嘴？

从我们对配置的定义是“无需修改源码就能改变应用行为的数据”不难看出，配置贯穿了业务/应用的所有层次，并且几乎可以囊括所有的元数据。因此，在谈及配置定义、配置系统实现和配置管理的时候，很容易牵扯到所有业务/应用设计问题，从而将整个软件工程都囊括了进来。

前文在讨论的时候，在配置的定义之外有一个隐含的假设，即我们关注的是需要人来管理的配置，而不是一般意义上的元数据。举个具体的例子，阿里巴巴开源的 [Nacos](https://nacos.io/) 也有人认为是配置管理系统，但它实际上脱胎于服务发现，而服务发现实际上是可以完全脱离人来管理的。对 Nacos 来说，“配置”首先是一个服务名到访问地址的映射。它的通用配置管理能力是后加的，这导致 Nacos 的通用配置管理缺少一系列基础的能力，例如环境管理和权限管理。虽然可以通过某些方式模拟，但是用起来总是非常别扭。

类似的，大部分能够想到的跟配置管理相关的软件，都杂糅了不同维度的需求。开发者在讨论的时候，往往基于一个具体的配置软件所实现的功能自由切换议题，而没有意识到他所讨论的实际上是完全不同的主题。

例如，Akka 的开发商 Lightbend 开源的 [lightbend/config](https://github.com/lightbend/config) 库就同时囊括了 HOCON 配置语言和不同配置来源的分层加载两种功能。

例如，CUE 在支持配置项相互引用之外，还实现了配置校验功能和执行用户自定义命令的 Command 功能。其中配置校验还算是配置定义领域一个不错的前沿探索，但是执行任意 Command 显然就是手里只有 CUE 这个软件，为了解决通用编程语言应该解决的问题，过度扩展软件范畴的结果。

例如，配置变更如何感知这个具体的技术实现问题，既可能在 Viper 即应用端独立解决，也可能在配置来自独立配置系统时，配合配置系统主动推送解决。甚至放在一个完整的运维体系里，它有可能是多系统配合的结果。某个开发者在讨论这个议题时，通常已经有具体是什么配置要推送的假设，甚至有具体的技术方案选型范围，结果一问出来却是一个很泛化的问题，参与讨论的人自然是盲人摸象，各执一词。

又例如，配置管理和权限管理的问题，前文介绍的就是一个非常干净的领域模型，但是很容易有开发者在讨论权限管理问题的时候混进来关于配置是否加密的问题，或者配置更新流程的问题。后面这两个可以说跟配置系统都没有关系，但不影响在“配置管理”这个大箩筐下，有人认为这是同一个范畴内的问题。

可惜的是，软件行业的从业者们对配置的理解确实就是这样的模糊。就连“配置”这个词在不同语境下可能指的是一个领域、一个值或者一个修改值的动作，都有很强的歧义。所以试图定义一个 one-for-all 的配置方案，最终会把自己给彻底绕进去。

例如，Google 上搜索 open-source config management 第一个返回的 [Configu](https://docs.configu.com/introduction/overview) 就惨不忍睹。它的基础是 Apollo 式的集中配置管理系统，但是 SDK 和 CLI 又同时支持从不同的 ConfigStore 里读数据，也就是 config-rs 那样的定位。结果最后这个软件设计出来，就是首先有一套配置在哪里的配置，然后才是应用需要读取配置本身。

## 环境变量传递配置是不是好实践？

先说结论：它可以是。

例如，要在 Cloudflare 部署一个网站，启动时加载配置的最简单方案就是在界面上设置环境变量。例如外部服务的连接串，调用服务所需的 APIKey 等等。对于这样的简单建站需求，搞一堆复杂的配置方案是没有意义的。实际上，Cloudflare 的这种方案解决的就是如何传递不适合进入到版本管理系统当中的配置的问题。简单且有效。

再例如，ScopeDB 启动时，上报应用监控数据的端点在某个具体的部署环境里，只有容器启动时才能知晓。这种情况下，传递配置的最佳方案就是通过命令行参数或者环境变量。

有人批评使用 `.env` 文件管理配置的方式，并将它和用环境变量管理配置等同起来。在我看来，用 `.env` 文件来管理配置，通过复制 `.env` 文件来应用不同的配置，当然是很危险且不可靠的方案。但是这跟用环境变量来传递配置其实并不等同。而对于 Cloudflare 独立建站这样一个简单的场景，本地搞个 .env 以仅在当前工作目录下才加载对应的环境变量，线上手动配置需要的变量，这是非常 OK 的。

## Single Source Of Truth 可行吗？

前文介绍当中提到，进程如何加载配置通常导向两个具体问题：

1. 配置信息从哪里获取？
2. 相同路径的配置项，以哪个为准？

其中第二点隐含了同一个配置项可能存在多个不同的来源。

在实践当中，我有以下两条经验。

其一，配置如果能做到 Single Source Of Truth 当然最好，毕竟这样一来在变更配置或分析问题的时候就很简单了。按照业务配置和应用配置的分类，业务配置应该全部都能做到单一来源，否则就很可能是工程设计有问题。进一步地，业务配置最好都来自同一个单一来源，最典型的就是独立配置系统，不管它是 Apollo 这样的专门系统，还是 GitOps 模拟的配置源，都可以。

其二，应用配置可能会有多个来源，但是多个来源的优先级应该是确定的。最典型的优先级从低到高的加载顺序就是：

1. 代码里的默认配置
2. 配置文件里的值
3. 环境变量的配置值
4. 命令行参数的配置值

其中，尽量不要在代码里提供默认 fallback 配置，取而代之地可以支持生成默认配置文件，而在加载配置时要求配置项必须存在。可选配置只需留给以下两种：不配置就禁用这个功能模块，或者绝大多数情况下可以由应用程序推导出来，例如线程池并发数。

这样，实践当中，应用启动时就加载，且贯彻整个应用生命周期的配置来源，就可以简化成只看配置文件里的值，以及极少数被环境变量覆盖的值。甚至对于确定会被环境变量覆盖的值，在配置文件里不再书写，进一步减少分析配置来源时的负担。之于命令行参数的配置项，我个人的经验是其配置值往往只在启动瞬间有效，或者永远只在启动时才知晓，尽量不要重复出现在其他来源。

终极解决方案是手写配置加载逻辑，给每一个配置项都打上来源标签，并在启动时打印日志说明最终配置值的来源。

至于从外部配置系统加载上来的应用配置，它的优先级比较模糊。通常，此类配置不应该重复出现在上面任何一个来源里，而是完全由外部系统管理，并且可能在不重启进程的前提下热更新。

有人在批评配置来源混乱时，关注点在配置可能出现在以下不同的来源：

* Apollo 配置中心
* AWS Parameter Store
* 数据库配置表
* 硬编码到代码
* LaunchDarkly 的 feature flags

可以看到，用我们上面的分类方式，这其实只有两种类型的来源，其一是硬编码的默认配置，我们已经说明大部分最好去掉。其二都是某种形式的外部系统，其实具体是哪个外部系统并不是那么重要。对于业务配置，尽量全是同一个来源。对于应用配置，尽量保证同一个配置只有一个来源。这就可以了。
