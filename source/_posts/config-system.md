去年底读到一篇题为[《配置管理：被忽视的技术债务》](https://mp.weixin.qq.com/s/BnmFuA7xb7F0oEhDmlmv5w)（下面简称《配置管理》）的文章，文章作者从产品研发的视角出发吐槽了配置管理的一系列问题：

* 太多 Source Of Truth 造成的混乱
* `.env` 文件管理配置缺乏一致性和安全性
* 现有的配置系统或服务都不能彻底解决作者所认为配置领域包含的问题

同时，文章作者提出了配置分类的几个维度，以及设计配置项时应该考虑的几个问题。

我于是发现，配置系统的设计和不同业务的配置管理问题，确实缺乏讨论。很少有开发者认真思考过配置系统是如何工作的，好像应用程序天然就能自己加载到正确的配置。一旦应用加载到错误的配置，开发者则往往很难说出问题出在哪里。要么怪罪配置有多个来源造成混乱，要么反思配置项的设计不够合理，要么指责配置管理职责不明，莫衷一是。

我有幸在某公司工作时看到过完整流畅的配置系统和管理方案。本文将综合业务研发和配置系统研发两种角度，首先给配置下一个定义，进而为市面上的配置软件做一个分类，最后再针对上面引用的文章提到的问题做逐一评论。

## 什么是配置？

我们首先给配置下一个定义：**配置（Configuration）是一种无需修改源码就能改变应用行为的数据。**

这个定义可以包含所有开发者能够想到的具体配置。例如，应用程序源码可以同时包含多种业务策略的实现逻辑，运行时通过配置业务功能开关选择启用或禁用特定策略。又例如，应用程序如果依赖数据库读写用户数据，数据库连接串通常也是作为配置项以某种形式传入应用程序的进程示例。

笼统地说，配置就是元数据，这是符合我们上面定义的。VSCode 编辑器可以配置个人偏好（Preference），macOS 有系统设置（Settings）。有人认为阿里巴巴开源的 Nacos 也是配置系统，但是它其实脱胎于服务发现。对 Nacos 而言，配置首先是一个服务名到访问地址的映射，这通常不需要人来手动管理，甚至人不能手动管理。还有人把 Pulumi 或 Ansible 也归类到配置系统里面，因为它们可以通过某种看起来非代码的形式改变基础设施系统的行为。

不过显然，把配置等同于所有元数据不利于讨论，因为上面提到的这种种配置实例明显对应着截然不同的心智模型。为了简化讨论，我们顺着《配置管理》的隐含假设，把范围限定在经典 Web 应用开发场景里，并且主要考虑需要人做确认的配置信息。

这初步解释了为何软件行业的从业者们在讨论配置管理问题时容易牛头不对马嘴。因为对配置共识的最大公约数是某种元数据，这就导致业务/应用所有层次的问题都可以关联到配置问题上，毕竟每一层都可以有自己的元数据。因此讨论配置问题，很容易牵扯到业务/应用设计问题，从而把整个软件工程都囊括了进来。具体讨论时，主题在软件工程的各个议题间自由切换，自然各执一词。

## 配置软件的分类

另一个导致讨论配置问题容易失焦的原因，则来源于现实世界配置软件的实现。目前存在的绝大部分配置管理相关的软件，都杂糅了不同维度的需求。从业者们在讨论配置问题的时候，往往不仅假设了具体场景和具体需求，还隐含的圈定了几个具体的配置软件的实现作为背景。实际讨论时，基于具体的配置软件所实现的功能任意切换议题，而没有意识到他所讨论的实际上是完全不同的主题。

我把配置软件的主要关注点总结为以下三个：

1. 如何书写配置
2. 如何加载配置
3. 如何管理配置

### 配置语言定义如何书写配置

最简单的配置表达方式是字符串键值对。例如通过环境变量配置连接串，可以写成 `export SCOPEDB_ENDPOINT="https://tenant-name.scopedb.cloud/"` 的形状。

进一步地，要想表达层次结构的配置，可以采用 JSON、YAML、TOML 等格式。形如：

```yaml
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-cloudflare
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: account@company.com
    privateKeySecretRef:
      name: letsencrypt-cloudflare-account-key
    solvers:
      - dns01:
          cloudflare:
            apiTokenSecretRef:
              name: <cloudflare-api-token-secret>
              key: api-token
```

再进一步，如果想要在一个配置项中引用另一个配置项，或者高度自定义某一配置项如何组装，可以使用 HOCON、CUE 或 Jsonnet 这样的复杂配置语言。以下是一个 Jsonnet 配置的例子：

```lua
local application = 'my-app';
local module = 'uwsgi_module';
local dir = '/var/www';
local permission = 644;

{
  'uwsgi.ini': std.manifestIni({
    sections: {
      uwsgi: {
        module: module,
        pythonpath: dir,
        socket: dir + '/uwsgi.sock',
        'chmod-socket': permission,
        callable: application,
        logto: '/var/log/uwsgi/uwsgi.log',
      },
    },
  }),
}
```

### 进程如何加载配置

第二类配置软件解决进程如何加载配置的问题。

不同编程语言生态都会有这样的软件库，但是往往并不非常流行，因为常见的做法是每个应用自己定义进程如何加载配置值。这里首先罗列不同语言生态当中的参考实现：

* Go 语言的实现可算是最流行的，其名为 [Viper](https://github.com/spf13/viper)
* Rust 语言的实现用户也不少，叫做 [config-rs](https://github.com/rust-cli/config-rs)
* Ruby 语言的实现叫 [config](https://github.com/rubyconfig/config)
* Java 语言也有类似的实现，但是知名度就要低许多，名字是 [cfg4j](https://github.com/cfg4j/cfg4j)
* Java 生态里更加常见的配置加载方案是 Spring Boot 内嵌的 [PropertySource](https://docs.spring.io/spring-boot/4.0/reference/features/external-config.html)

所有这类软件解决的都是同样的两个核心问题：

1. 配置信息从哪里获取？
2. 相同路径的配置项，以哪个为准？

可以看到，上面所举的示例软件，大多支持环境变量、命令行参数、不同格式的配置文件、远程配置服务等配置信息来源。在软件内部，它们都会将所有加载上来的配置信息解析成统一的层次配置结构。对于相同路径的配置项，这些软件大多有预先定义的加载优先级，部分软件可以自定义加载优先级。

现实世界当中，配置来源千奇百怪，加载优先级跟具体技术实现应用需求强挂钩。上面介绍的开箱即用方案往往不能很好的理解应用需求，导致其预制逻辑经常出乎开发者预料。再加上从不同来源读取配置信息的库往往是现成的，实现一个分层加载方案并不复杂，因此，一旦预制逻辑不能满足应用需求，大部分应用程序会绕过这类配置库，直接内联实现配置加载逻辑。不过，自己实现的配置加载逻辑仍然需要解决上面两个问题。

### 配置的环境和权限管理

最后一类配置软件关注配置的环境和权限管理。

由于其功能的复杂性，这类软件往往会实现会以独立进程的形式运行，并提供 SDK 给应用软件执行管理操作。

典型的配置系统领域模型大致是：

* 一个组织可以有多个团队和成员，一个成员可以属于若干个团队
* 一个组织可以有多个服务，每个服务对应到一个线上业务或微服务
* 一个服务可以有多个环境，每个环境各自有一套配置信息

对于权限管理，则有两个维度：

* 某个成员是否有某个服务在某个环境下的所有配置的读或写权限
* 某个服务是否有读另一个服务在相同环境下的配置的权限

其他配置管理系统也基本大同小异。

### 其他配置管理的需求

至此，我们讨论了三种配置系统主要的关注点。然而，在现实世界当中，实际读写配置还有大量细节问题或外围问题。例如配置更新的流程如何设计，配置存储如何选择，配置变更后如何推送，配置如何验证，私有配置如何加密等等。我们没有单独分类，是因为这些问题的解决方案，要么是上述几种软件的一个具体模块，要么对应的软件并非专为配置问题设计的。

## 业务配置 vs. 应用配置

配置的分类可以有很多维度。如果从中选一个最重要的维度，我会选择问这一个问题：**这是一个业务配置，还是一个应用配置？**

业务配置指的是设计上与应用实现无关，只影响业务逻辑的配置。例如业务功能开关，同类活动可配置用户标签，市场关键词列表，折扣策略对照表等等。

应用配置则指的是控制应用程序自身运行，无关业务逻辑的配置。例如监听端口，数据库连接串，系统日志的输出等级等等。

这个分类首先解决了权责划分问题，**对于业务配置来说，有权修改配置的是产品和运营；而对于应用配置来说，有权修改配置的是研发和运维。**

当然，实际组织运行时，研发有可能作为产品和运营的代理执行配置修改动作，DevOps 可能是一个人同时担任，甚至小团队里面，所有的职责可能都对应到同一个人。但是这不影响软件工程视角下的责权划分。

明确责任以后，具体的技术选型反而非常灵活。当然，业务配置和应用配置的技术实现有一定的倾向性，但都不绝对。

例如，业务配置往往是简单的键值对，不需要层次结构。即使是前面提到的市场关键词列表和折扣策略对照表这样的配置，也是一层键值映射，只是配置值可能有不同的类型；应用配置则经常有层次结构，以指定不同模块所需的信息或具体行为策略。

对于配置校验，业务配置的校验往往写在代码里随新版本发布，校验规则相对随机；应用配置的校验规则相对明确，可以受益于校验工具。

再看配置值的来源，业务配置更可能来自独立配置系统，也更容易有配置热更新的需求；应用配置则更可能来自配置文件、命令行参数、环境变量等，且通常在启动后就不再变更，或者说更新通常需要一次应用重启。

## 《配置管理》问题解答

### 环境变量传递配置是不是好实践？

先说结论：它可以是。

例如，要在 Cloudflare 部署一个网站，启动时加载配置的最简单方案就是在界面上设置环境变量。例如外部服务的连接串，调用服务所需的 APIKey 等等。对于这样的简单建站需求，搞一堆复杂的配置方案是没有意义的。实际上，Cloudflare 的这种方案解决的就是如何传递不适合进入到版本管理系统当中的配置的问题。简单且有效。

再例如，ScopeDB 启动时，上报应用监控数据的端点在某个具体的部署环境里，只有容器启动时才能知晓。这种情况下，传递配置的最佳方案就是通过命令行参数或者环境变量。

《配置管理》批评使用 `.env` 文件管理配置的方式，并将它和用环境变量管理配置等同起来。

在我看来，用 `.env` 文件来管理配置，通过复制 `.env` 文件来应用不同的配置，当然是很危险且不可靠的方案。但是这跟用环境变量来传递配置并不等同。而对于 Cloudflare 独立建站这样一个简单的场景，本地搞个 `.env` 以仅在当前工作目录下才加载对应的环境变量，线上手动配置需要的变量，这是完全说得通的。

### Single Source Of Truth 可行吗？

前文介绍当中提到，进程如何加载配置通常导向两个具体问题：

1. 配置信息从哪里获取？
2. 相同路径的配置项，以哪个为准？

其中第二点隐含了同一个配置项可能存在多个不同的来源。

在实践当中，我有以下两条经验。

其一，配置如果能做到 Single Source Of Truth 当然最好，毕竟这样一来在变更配置或分析问题的时候就很简单了。按照业务配置和应用配置的分类，业务配置应该全部都能做到单一来源，否则就很可能是工程设计有问题。进一步地，业务配置最好都来自同一个单一来源，最典型的就是独立配置系统，不管它是 Apollo 这样的专门系统，还是 GitOps 模拟的配置源，都可以。

其二，应用配置可能会有多个来源，但是多个来源的优先级应该是确定的。最典型的优先级从低到高的加载顺序就是：

1. 代码里的默认配置
2. 配置文件里的值
3. 环境变量的配置值
4. 命令行参数的配置值

其中，尽量不要在代码里提供默认 fallback 配置，取而代之地可以支持生成默认配置文件，而在加载配置时要求配置项必须存在。可选配置只需留给以下两种：不配置就禁用这个功能模块，或者绝大多数情况下可以由应用程序推导出来，例如线程池并发数。

这样，实践当中，应用启动时就加载，且贯彻整个应用生命周期的配置来源，就可以简化成只看配置文件里的值，以及极少数被环境变量覆盖的值。甚至对于确定会被环境变量覆盖的值，在配置文件里不再书写，进一步减少分析配置来源时的负担。至于通过命令行参数传递的配置项，我个人的经验是其配置值往往只在启动瞬间有效，或者永远只在启动时才知晓，尽量不要重复出现在其他来源。

终极解决方案是手写配置加载逻辑，给每一个配置项都打上来源标签，并在启动时打印日志说明最终配置值的来源。

至于从外部配置系统加载上来的应用配置，它的优先级比较模糊。通常，此类配置不应该重复出现在上面任何一个来源里，而是完全由外部系统管理，并且可能在不重启进程的前提下热更新。

《配置管理》在批评配置来源混乱时，关注点在配置可能出现在以下不同的来源：

* Apollo 配置中心
* AWS Parameter Store
* 数据库配置表
* 硬编码到代码
* LaunchDarkly 的 Feature Flags

可以看到，用我们上面的分类方式，这其实只有两种类型的来源。

一种是硬编码的配置。如果是默认配置，我们已经说明大部分最好去掉。如果就是写在代码里的值，外部不可覆盖，其实不符合我们前面提到的配置定义。这也是有可能的，很多运行常数并不必要暴露给用户配置。等到你有不改源码也能更新的需求时，它就变成了默认配置，这时在走上面的经验既可。

其他的都是某种形式的外部系统，其实具体是哪个外部系统并不是那么重要。对于业务配置，尽量全是同一个来源。对于应用配置，尽量保证同一个配置只有一个来源。这就可以了。

### 为何现有配置系统都不能解决配置领域内的所有问题？

《配置管理》点评了一系列文章作者认知下的配置系统：

* AWS Parameter Store
* Apollo
* Nacos
* LaunchDarkly

批评的点集中在以下两个问题：

1. 流程管理，主要是审批和审计
2. 成本问题，商业产品的价格和开源系统的维护

我们先从 LaunchDarkly 入手。《配置管理》声称 LaunchDarkly 解决的是 Feature Flags 的问题而不是配置管理的问题，而使用 LaunchDarkly 的主要痛点是价格太贵。

根据前面的讨论，很显然 Feature Flags 属于配置的一部分。《配置管理》所说的，那些不需要频繁更新的配置项，用 LaunchDarkly 太贵，但是抛开价格不谈，LaunchDarkly 的环境管理，配置变更审计和权限管理，灰度发布的能力，其实也没有给其他配置项带来什么麻烦。这些能力是不频繁更新的配置项所需能力的超集。

当然，LaunchDarkly 针对 Feature Flags 做了很多特化，强假设你是在管理 Feature Flags 并突出了 A/B 测试的流程和反馈。但你首先不能把它开除出配置管理的范畴里。至于为什么 LaunchDarkly 要做成一个 Feature Flags 特化的系统而不是通用配置系统，这当然是因为高频更新、灰度发布和 A/B 测试有商业价值了。如同《配置管理》自己提到的，荷兰市场的关键词这个配置你一个月更新一次，搞个 GitOps 拖拉机也能用，我做一个产品解决这个需求，卖多少钱你会买？

Nacos 前面已经聊过了，是个非典型配置管理系统。我倒觉得可以把 Nacos 开除出配置系统的范畴，把它当成某种版本化服务发现系统就行，基本不需要人手动配置。

Apollo 和 AWS Parameter Store 是比较经典的配置管理系统，我们复习一下这类系统基础的建模方式：

* 一个组织可以有多个团队和成员，一个成员可以属于若干个团队
* 一个组织可以有多个服务，每个服务对应到一个线上业务或微服务
* 一个服务可以有多个环境，每个环境各自有一套配置信息

对于权限管理，则有两个维度：

* 某个成员是否有某个服务在某个环境下的所有配置的读或写权限
* 某个服务是否有读另一个服务在相同环境下的配置的权限

以《配置管理》批评的两个点而言，价格这个真没什么好说的，你要么自己维护，要么去买一个。以现在 Agentic Coding 的实力，其实如果对配置系统的建模比较了然于心，完全可以基于云数据库自己搭一个方案出来。Apollo 的问题主要还是太老了，所以它的高可用方案比较复杂，云上的话，其实云数据库 + Kubernetes Deployment 就解决了。

接下来说重点说一下功能集成和流程集成的问题。

《配置管理》批评 Apollo 不支持自定义加密密钥，那你 fork 加一下或者硬编码改成你要的不就完了😅

《配置管理》批评审计问题。其实审计是非常好做的，就是操作日志找个地方存起来就好了。然后面向审计场景，在知道日志格式的情况下，把重要的操作日志单独拎出来，用视觉友好的方式渲染一下，结束。为什么 Apollo 不好做呢？有两个原因。

第一个是因为没人做。其实社区是有 issue 的，甚至 Apollo 的领域模型设计里已经存了操作日志，只是没做上面提到的后面两步。开源软件嘛，你有需求就 Agentic Coding 补一下呗。

* https://github.com/apolloconfig/apollo/issues/3505
* https://github.com/apolloconfig/apollo/issues/5457

直接实现你公司里的需求都非常简单，难的是做通用。而通用在这里可能是没有意义的。

第二个原因是因为公司的环境一般是高度集成的。你在开源的范畴里做，你得支持审计日志存到不同的地方，还有不同的展示方式吧？但是公司的审计需求一般跟自己的流程和组织架构强相关，而且不需要插件化，就用特定的选择就行。根据我的经验，开源方案内用都会有严重的阻抗失配问题，都是要二次开发的。再看 AWS Parameter Store 同样可以把操作日志导出到 CloudTrail 或者 S3 里面，只不过同样后面这个展示的部分没做好，因为不同公司的审计需求千奇百怪。

权限集成问题涉及到跟公司 OA 系统集成，OA 系统是没有公开标准的，而且也不知道你的组织架构长啥样。流程集成问题类似，一般公司会有一个独立的流程审批管理系统，在那个系统里反过来调用配置管理系统更新配置是更合理的，而不是配置系统自己定义一个流程审批功能。

我在某公司（下称 T 公司）使用的完整配置管理方案（下称 L 系统），其中独立配置服务也支持以下功能：

* 权限分组管理（面向人）
* 配置变更审计
* 系统操作日志
* 配置访问控制（面向服务）
* 灰度发布管理
* 密码配置自动加密

读取该独立配置服务的 SDK 支持配置热更新。

L 系统在 T 公司内大概就 2-3 个人维护，这还是五年前的水平。实际上一个人带上 AI 兼职维护，控制一下需求也绰绰有余。

其中，L 系统的环境管理是和企业持续部署（CD）高度集成的。整个企业就是特定的几个环境（测试 + N 个线上），CD 系统和配置环境名称一一对应。在开源方案或者第三方系统里，基本很难做到这个，你得适应不同用户的需求。

例如，为了支持在具体部署环境里进一步区分不同配置 Profile（主要是测试环境里不同研发开发分支需要不同配置，线上环境预发时不同逻辑，A/B 测试等等），L 系统在环境下面添加了泳道的概念。这大致对应 Apollo 的环境/命名空间划分。但是 Apollo 在环境和命名空间之间还区分了不同的集群，例如线上环境/上海 A 集群，而这在 L 系统里被铺平成多个环境，对应线上上海 A 环境。到了 AWS Parameter Store 里，这些全部需要通过顶层的 tag 概念来模拟。此外，Apollo 的每个应用都可以有不同的集群和命名空间，而 L 系统全局是统一的几个环境。这些概念都会深刻影响公司里 CD 系统的领域模型。

至于权限，Apollo 的权限管理设计是：

* 应用属于某个部门
* 每个应用可以指定唯一负责人，具有该应用所有配置的所有权限
* 负责人可以授权任何人对最细到命名空间粒度的所有配置读或写权限

AWS Parameter Store 自然是跟 IAM 紧密结合，详细到 Parameter Keys 粒度给某个身份授予读或写权限。什么部门、应用、命名空间，自己想办法建模吧。

L 系统跟 T 公司的 OA 系统是深度集成的。首先部门信息就是从 OA 系统拉进来的，不需要手动录入或注册。进一步地，其权限模型大致是：

* 一个微服务对应一个项目
* 项目有项目负责人和项目成员
* 项目成员只能查看和修改测试环境的配置，不能查看更不能修改线上环境的配置
* 项目负责人进一步的可以查看线上环境的配置，可以修改线上公开配置，可以添加项目成员
* 项目绑定业务，业务从属于团队，例如用户团队有用户登录业务，支持用户登录有微服务项目 A、B、C 等等
* 项目绑定的业务需指定业务负责人，可以添加项目负责人和项目成员，业务负责人可以修改线上非公开配置
* 项目绑定的团队需指定团队负责人，团队负责人进一步可以添加业务负责人
* 业务负责人可以创建项目

具体到查看审计日志、灰度管理，配置访问控制，还有进一步的权限对应关系。一般来说，团队和业务负责人就是对应组织架构里大团队/业务团队的老大，项目负责人产品和研发都可以担任或者共同担任。对于 T 公司来说，L 系统的领域模型可以完美对应到组织架构，而想要找一个第三方系统这么严丝合缝，就很难了。

至于流程集成，Apollo 和 Parameter Store 都不管流程。原因前面提到了，相比让配置系统内置流程管理，更合理的方案是让流程管理系统调用配置系统接口更新。

L 系统的变更不需要跟流程系统集成，因为前面的权限划分和审计已经搞定了所有流程需要的细节：永远只有具有审批权的人才能操作，所有操作都有审计日志。由于配置管理的权限大部分下放到项目负责人这一层，所以项目团队（通常是五人以内的小组 + PM）可以自己闭环整个流程，自然也就不需要额外审批。

当然，T 公司也有流程审批系统，恰好我维护过一段时间。它的核心也很简单，就是一个审批流。每个审批流程对应一个固定流程图，流程图上有若干个审批节点和自动条件检查。当审批通过时自动执行请求操作或推送到执行人等待其操作后确认。研发生命周期里，主要用于数据库变更审批。

有了 L 系统以后，在 T 公司开发微服务，最后配置加载情况如下：

1. 业务配置基本都从 L 系统加载
2. 应用配置有些写在 Spring Boot 打包发布的 application.properties 文件里
3. 密钥配置也用 L 系统管理，少数特定性能参数也会用 L 系统管理，以实现热更新
4. 绝大部分外部依赖的连接串等，要么由对应系统提供的 SDK 直接根据服务名查找注入，要么是对应系统集成持续部署系统在启动时自动传参
5. 本地开发时可以通过命令行参数或本地的配置文件做覆盖

回到一开始的问题，为何现有配置系统都不能解决配置领域内的所有问题？因为简单的功能做了买不上价，复杂的功能跟公司流程和组织结构强绑定，做得太具体大部分公司一看跟组织架构或现有选型不一样，概念根本映射不过去，除非新的实践大行其道，反过来影响组织架构和选型；做得太通用就是 AWS Parameter Store 啥都要用户额外设计或者外挂。

所以，现实世界里存在的配置软件，要么只提供基础能力，不做复杂管理，要么专攻某个特定的配置场景，例如 A/B 测试这样可以直接检验商业收益的。除此之外，大家要么是现有方案凑合着用，反正公司不大没什么要管理，要么是自己写配置管理方案。而配置管理方案需要考虑的点，上面大都提到了。
